## 1. 说明

本文按真实用户功能流程说明数据如何流转。这里的“场景”不是单个原子操作，而是用户完成一个实际功能的完整过程。

不覆盖：

- 用户注册 / 登录。
- 购买积分。
- 订单支付。
- 订阅管理。

特别规则：

- 凡是调用 AI 的地方，都必须先进入 Redis 分布式锁。
- 凡是调用 AI 且消耗积分的地方，都必须执行积分预扣减。
- 积分预扣减操作的是现有 `Credit-积分余额` 和 `CreditAuditLog-积分流水`。
- AI 异步任务失败时，需要根据失败阶段执行积分补偿。

## 2. 同步与异步标记

| 标记 | 含义 |
| --- | --- |
| 同步 | 用户请求内立即完成，接口返回前完成 |
| 异步 | 接口先返回任务或状态，后续由 Worker / Queue / Provider callback 完成 |
| 锁 | 使用 Redis 分布式锁保护并发 |
| 预扣减 | AI 调用前先扣减积分 |
| 补偿 | AI 失败后按规则退回积分或标记失败 |

## 3. 场景 A：用户提交主题想法

用户在站内提交一个每日主题想法。该流程不调用 AI 时可以完全同步；如果启用自动质量/安全审核，则审核部分可异步。

涉及数据表：

- `User-用户`
- `ThemeSubmission-主题提交`
- `ReviewRecord-审核记录`
- `Credit-积分余额`
- `CreditAuditLog-积分流水`

用户在 Home 输入 prompt，选择模型、风格、比例和数量，点击 Generate。该流程调用 AI，需要 Redis 分布式锁和积分预扣减。

- `User-用户`
- `Credit-积分余额`
- `CreditAuditLog-积分流水`
- `CreationSession-创作会话`
- `GenerationJob-生成任务`
- `GeneratedImage-生成图片`
- `ReviewRecord-审核记录`

## 6. 场景 D：用户使用主题灵感后生成图片

所有 AI 场景统一遵循以下顺序：

<Mermaid
  title="13. AI 场景通用事务原则"
  chart={`
flowchart TD
  A[收到用户 AI 请求] --> B[Redis 分布式锁]
  B --> C[校验用户权限和资源归属]
  C --> D[估算积分成本]
  D --> E[预扣减 Credit-积分余额]
  E --> F[写入 CreditAuditLog-积分流水]
  F --> G[创建业务任务或日志]
  G --> H[投递异步任务]
  H --> I[释放 Redis 锁]
  I --> J[Worker 调用 AI]
  J --> K{AI 是否成功}
  K -->|成功| L[写入结果表并更新任务成功]
  K -->|失败且需退款| M[补偿积分并写流水]
  K -->|失败但不退款| N[记录失败原因]
`}
/>

必须保证：

- 同一个用户同一个请求不能重复扣积分。
- 预扣减和任务创建要有可追溯关联。
- `CreditAuditLog.operationReferId` 必须指向业务任务 ID。
- Worker 重试必须幂等。
- AI 结果写入必须幂等，避免重复创建图片。
