## 1. 说明

本文按真实用户功能流程说明数据如何流转。这里的“场景”不是单个原子操作，而是用户完成一个实际功能的完整过程。

不覆盖：

- 用户注册 / 登录。
- 购买积分。
- 订单支付。
- 订阅管理。

特别规则：

- 凡是调用 AI 的地方，都必须先进入 Redis 分布式锁。
- 凡是调用 AI 且消耗积分的地方，都必须执行积分预扣减。
- 积分预扣减操作的是现有 `Credit-积分余额` 和 `CreditAuditLog-积分流水`。
- AI 异步任务失败、被安全策略拦截、供应商无结果或超时时，必须由积分服务完成返还，最终业务语义是不消耗积分。

## 2. 同步与异步标记

| 标记 | 含义 |
| --- | --- |
| 同步 | 用户请求内立即完成，接口返回前完成 |
| 异步 | 接口先返回任务或状态，后续由 Worker / Queue / Provider callback 完成 |
| 锁 | 使用 Redis 分布式锁保护并发 |
| 预扣减 | AI 调用前先扣减或占用积分 |
| 返还 | AI 失败后由积分服务按 `operationReferId` 幂等返还预扣减积分 |

## 3. 场景 A：用户创建并提交主题提案

用户在站内创建每日主题提案。提交前是草稿，可以编辑；提交后进入审核，普通用户不能再修改。该流程不调用 AI 时可以完全同步；如果启用自动质量/安全审核，则审核部分可异步。

涉及数据表：

- `User-用户`
- `ThemeSubmission-主题提交`
- `ReviewRecord-审核记录`
- `Credit-积分余额`
- `CreditAuditLog-积分流水`

- 普通用户只能编辑自己的 `draft`。

## 5. 场景 C：用户在首页自由生成图片

用户在 Home 输入 prompt，选择模型、风格、比例和数量，点击 Generate。该流程调用 AI，需要 Redis 分布式锁、积分预扣减和异步任务状态轮询。

涉及数据表：

- `User-用户`
- `Credit-积分余额`
- `CreditAuditLog-积分流水`
- `GenerationJob-生成任务`
- `GeneratedImage-生成图片`
- `ReviewRecord-审核记录`

关键状态变化：

- `GenerationJob.status`: `pending -> queued -> running -> succeeded / failed / blocked`
- `GenerationJob.estimatedCredits`: 创建任务时记录预计消耗，用于展示和对账
- `GenerationJob.chargedCredits`: 成功时记录实际消耗；失败、拦截或取消时为 `0`
- `GeneratedImage.status`: `generated`
- `CreditAuditLog.operationReferId`: `GenerationJob.jobId`
- `SafetyService` 是固定扩展入口；MVP 可返回 `skipped` 或轻量 `passed`，后续可替换为自研 NSFW 审查或第三方审核
- `SafetyService.status = skipped` 表示当前阶段未启用复杂审查，不阻断流程；`blocked` 或 `provider_rejected` 才会终止任务并返还积分

## 6. 场景 D：用户使用主题灵感后生成图片

用户在 Theme Detail 或 Studio 中选择一个主题，点击 `Get ideas`，使用某个 idea 填入 prompt，再点击 Generate。

该流程有两个 AI 触点：

- Assistant 生成 ideas。
- 图片生成。

当前版本 Assistant 中间过程不持久化，ideas 只作为前端临时状态。只有用户最终点击 Generate 的 prompt 写入 `GenerationJob`。如果后续 Assistant 需要收费，应单独设计 Assistant 任务或脱敏日志，不复用图片生成任务。

涉及数据表：

- `User-用户`
- `Theme-正式主题`
- `Credit-积分余额`
- `CreditAuditLog-积分流水`
- `GenerationJob-生成任务`
- `GeneratedImage-生成图片`

关键状态变化：

- `ReferenceImage.safetyStatus`: `pending -> passed / skipped / failed`
- `GenerationJob.referenceId`: 关联参考图
- `GeneratedImage` 是结果作品，`ReferenceImage` 仍只是输入素材
- QStash 任务消息只传 `jobId`，不传 reference image URL 或 base64 图片
- OpenRouter 输出如果是 base64 data URL，必须先落 R2，再保存 `GeneratedImage`
- MVP 阶段 `SafetyService` 可以只做文件基础校验并返回 `skipped`；后续接入图片 NSFW 审查时不需要改变上传和生成主流程
- `SafetyService.status = skipped` 不阻断流程；`blocked` 会让生成任务进入 `blocked` 并触发积分返还

- `User-用户`

- `Theme-正式主题`
- `ImageSubmission-作品投稿`
- `ReviewRecord-审核记录`

  Review->>DB: 异步写入 ReviewRecord(theme_relevance)
  Review->>DB: 异步更新 ImageSubmission(status=needs_review/published/rejected)

  alt 审核通过
    Review->>DB: 创建 PublicImage-公开图片
    Review->>DB: 更新 GeneratedImage(status=published)
  else 审核拒绝
    Review->>DB: 更新 GeneratedImage(status=rejected)
  end
`}
/>

关键状态变化：

- `ImageSubmission.status`: `under_review -> needs_review / published / rejected`
- `GeneratedImage.status`: `generated -> under_review -> published / rejected`
- `PublicImage` 只在审核通过后创建

关键约束：

- 投稿目标是 `Theme-正式主题`，不能是 `ThemeSubmission-主题提交`。
- `ImageSubmission-作品投稿` 不改变 `ThemeSubmission-主题提交` 状态。
- 如果用户想基于某张图片提出新主题，应走场景 J 中的“从图片创建主题提案”入口。

## 13. AI 场景通用事务原则

所有 AI 场景统一遵循以下顺序：

<Mermaid
  title="13. AI 场景通用事务原则"
  chart={`
flowchart TD
  A[收到用户 AI 请求] --> B[Redis 分布式锁]
  B --> C[校验用户权限和资源归属]
  C --> D[估算积分成本]
  D --> E[预扣减 Credit-积分余额]
  E --> F[创建业务任务或日志]
  F --> G[投递异步任务]
  G --> H[释放 Redis 锁]
  H --> I[前端轮询任务状态]
  H --> J[Worker 调用 AI]
  J --> K{AI 是否成功且有可交付结果}
  K -->|成功| L[写入结果表并更新任务成功]
  L --> M[记录 chargedCredits=estimatedCredits]
  K -->|失败| N[记录失败原因并更新任务失败]
  N --> O[积分服务按 jobId 幂等返还预扣减积分]
`}
/>

必须保证：

- 同一个用户同一个请求不能重复扣积分。
- 预扣减和任务创建要有可追溯关联。
- `CreditAuditLog.operationReferId` 必须指向业务任务 ID。
- `GenerationJob` 不承载积分状态机，只通过 `estimatedCredits` 和 `chargedCredits` 辅助对账。
- 失败返还由积分服务保证幂等和最终成功。
- Worker 重试必须幂等。
- AI 结果写入必须幂等，避免重复创建图片。
