## 1. 设计边界

本文基于 [Monica-AI.md](./Monica-AI.md) 输出后端技术方案，覆盖：

- 后端模块划分。
- 新增业务数据 Model。
- API 职责、鉴权和权限边界。
- 生成任务、投稿审核、Studio、Explore 的状态流转。
- 与现有用户、积分、订单系统的集成方式。

不覆盖：

- AI prompt 模板、模型调用参数、供应商 SDK 细节。
- i18n 方案。
- 前端组件设计和样式。
- 管理后台完整交互设计。

AI 调用相关逻辑在本文中只保留接口占位。

## 2. 现有基础能力

当前项目已经存在以下基础 Model：

- `User`
- `Credit`
- `CreditAuditLog`
- `Subscription`
- `Transaction`
- `UserBackup`
- `Apilog`

这些 Model 不重新设计。

业务功能需要复用现有能力：

- 用户身份：使用 `User.userId` 作为业务外键。
- 登录鉴权：使用 `ApiAuthUtils`。
- 积分余额：使用现有 `Credit`。
- 积分流水：使用现有 `CreditAuditLog`。
- 订单和订阅：使用现有 `Transaction` / `Subscription`。

API Route 中鉴权统一使用：

```ts
import { ApiAuthUtils } from '@windrun-huaiin/backend-core/auth/server';

const authUtils = new ApiAuthUtils(request);
const { user } = await authUtils.requireAuthWithUser();
```

可选登录态使用：

```ts
const authUtils = new ApiAuthUtils(request);
const auth = await authUtils.getUser();
```

### 3.4 OpenRouter 图片输入输出边界

后续 AI model 统一通过 OpenRouter 接入时，图片输入输出必须按以下边界处理：

- 图文生图的输入图片可以使用可访问 URL，也可以使用 base64 data URL。
- 用户上传的 reference image 默认先落到 R2，再在 Worker 调用 AI 前生成可被 OpenRouter / Provider 访问的 public 或 signed view URL。
- signed URL 的有效期必须覆盖 QStash 排队时间、Worker 执行时间、Provider 拉图时间和必要重试窗口。
- 不把 `protected.view` 这类需要业务鉴权的 URL 传给 OpenRouter，因为第三方 Provider 不会携带站内鉴权 header。
- 如果某些图片必须绝对私有，可在 Worker 内转成 base64 data URL 传入，但这不是默认策略。

OpenRouter 图像生成输出按官方文档应按 `message.images[]` 处理。每个 image item 里的 `image_url.url` 可能是 base64 data URL，而不是可长期访问的公网 HTTP URL。

因此生成结果保存规则是：

1. Worker 从 OpenRouter response 中读取 `message.images[]`。
2. 遍历每个 `image.image_url.url`。
3. 如果是 `data:image/...;base64,...`，解析 mime type 和 base64 内容。
4. 将图片二进制上传到 R2。
5. 保存 `GeneratedImage.sourceImageUrl`，如果结果已迁移到 R2，则同时保存 `cdnImagePrefix`、`storageKey` 和尺寸。
6. 不把 OpenRouter 返回的完整 base64 或原始响应写入 `GenerationJob`；如需排查，仅写入脱敏后的 `Apilog` 摘要。

一次生成多张图片时，Worker 按 `message.images[]` 创建多条 `GeneratedImage`。不要假设所有模型都支持多图输出，具体数量以模型能力和实际响应为准。

## 4. 新增数据 Model 设计

### 4.1 命名原则

- 业务表全部放在 `monica-ai` schema。
- 主键使用 `BigInt @id @default(autoincrement())`，延续现有风格。
- 简单业务数据优先只使用 BigInt 主键，例如 `Theme`、`ThemeSubmission`。
- 会出现在图片 URL、任务轮询、公开图片分享或外部引用中的数据，可额外保留 `xxxId String @db.Uuid`，例如 `GenerationJob`、`GeneratedImage`、`PublicImage`。
- 与现有用户关联统一保存 `userId String @db.Uuid` 标量字段；Monica AI 业务表不在 `User` Model 上声明反向 relation，需要用户信息时由 service 显式查询。
- Monica AI 业务表之间不声明 Prisma relation，只保存 scalar id 和索引；列表页需要扩展展示信息时，由 repository/service 按 id 集合显式查询并组装。来源追溯字段如 `Theme.sourceSubmissionId`、`PublicImage.sourceSubmissionId` 只作为标量查询条件。
- 状态字段使用 `String @db.VarChar(50)`，不使用 Prisma enum，保持和现有 `User.status`、`Transaction.orderStatus` 一致。
- 审核详情、生成结果图片元数据等非核心展示字段可使用 `Json?`。
- 供应商原始 request/response 不进入业务表；如需排查，写入脱敏后的 `Apilog` 或外部日志系统。
- 删除优先使用 `deleted Int @default(0)` 或状态隐藏，不物理删除业务关键记录。

新增 Model：

- `ThemeSubmission`
- `Theme`
- `ReferenceImage`
- `GenerationJob`
- `GeneratedImage`
- `ImageSubmission`
- `PublicImage`
- `ImageLike`

### 4.3 Model 建立理由

<CheetTable title="Model 建立理由">

| Model | 建立理由 |
| --- | --- |
| `ThemeSubmission` | 承接用户提交的主题文本想法，保存原始标题、详情、提交理由、审核状态、用户侧 Notes、采纳结果和奖励发放依据 |
| `Theme` | 表示管理员创建或采纳后的正式主题，独立保存前台展示、SEO、生成器灵感、发布日期、精选图片和统计字段；普通用户不能直接创建 |
| `ReferenceImage` | 保存用户上传的参考图输入，负责归属、安全审核和被生成任务引用 |
| `GenerationJob` | 记录图片生成任务，保存最终生成参数、异步状态、预计/实际消耗、失败原因和结果回写；Provider/QStash/锁细节不进入业务表 |
| `GeneratedImage` | 保存 AI 生成出的用户私有图片资产，是 My Studio、下载和投稿的核心 |
| `ImageSubmission` | 表示用户将私有生成图提交到某个主题的候选作品，保存公开英文标题、创作说明、当前审核状态和 `reviewFlow` 审批流节点，支撑 Submitted、Approved、Rejected |
| `PublicImage` | 表示已发布图片的公开展示投影，单独承载点赞数、收藏数、精选分、公开状态等展示层数据 |
| `ImageLike` | 记录用户点赞关系，支持取消点赞、当前用户 liked 状态和排序统计 |
| `ImageSave` | 记录用户收藏关系，支持取消收藏、当前用户 saved 状态和个人收藏扩展 |

</CheetTable>

### 4.4 图片类 Model 边界

图片相关数据不应该混成一张大表。当前设计中有 3 个和图片直接相关的 Model：

<CheetTable title="图片类 Model 边界">

| Model | 是否存储图片资产 | 来源 | 主要用途 |
| --- | --- | --- | --- |
| `ReferenceImage` | 是 | 用户本地上传 | 作为生成参考图，归属于用户和创作会话，需要安全审核 |
| `GeneratedImage` | 是 | AI 生成结果 | 用户私有图片资产，服务 My Studio、下载和投稿 |
| `PublicImage` | 否，不重复存图 | `GeneratedImage` 投稿审核通过后创建 | 公开展示投影，服务 Explore、Theme Gallery、点赞、收藏、精选排序 |

</CheetTable>

关键规则：

- `ReferenceImage` 是输入素材，不是作品，不进入 Studio Gallery，也不能直接投稿。
- `GeneratedImage` 是生成结果，是 Studio 的核心图片资产。
- `PublicImage` 不是另一份图片文件，它引用 `GeneratedImage.imageId`，只保存公开展示层需要的数据，如 `likeCount`、`saveCount`、`viewCount`、`uniqueViewCount`、`featuredScore`、`promptPublic`。
- 用户自己本地上传的图片如果只是参考图，进入 `ReferenceImage`。
- 后续如果支持“用户上传自己的作品参与投稿”，再单独讨论是否新增 `UploadedImage`。当前产品原型是 AI 图片创作，投稿对象默认来自 AI 生成结果。

### 4.5 主题类 Model 边界

主题相关数据分为“提案”和“正式主题”两层：

<CheetTable title="主题类 Model 边界">

| Model | 谁创建 | 是否公开展示 | 主要用途 |
| --- | --- | --- | --- |
| `ThemeSubmission` | 普通用户提交 | 否，仅提交人和管理员可见 | 用户原始主题想法、审核结果、用户侧 Notes、采纳来源和积分奖励依据 |
| `Theme` | 管理员直接创建，或接受用户提交后创建 | 是，按发布日期和状态展示 | 今日主题、历史主题、创作上下文、生成器灵感、主题 Gallery、SEO 和作品投稿目标 |

</CheetTable>

关键规则：

- 普通用户不能直接创建 `Theme`。
- 普通用户提交主题时只写入 `ThemeSubmission.title`、`details`、`submitReason`。
- `ThemeSubmission` 不保存正式主题的 SEO、tags、generator ideas、featured images，也不保存管理员编辑后的正式标题/简介/说明。
- 管理员可以审核所有 `ThemeSubmission`，接受后创建一条独立的正式 `Theme`，由 `Theme.sourceSubmissionId` 指向原提交。
- 管理员也可以在 `Manage themes` 直接创建 `Theme`，不需要先伪造一条用户提交。
- 正式 `Theme` 的字段可由管理员基于提交内容初始化，但后续允许独立修改，不和 `ThemeSubmission` 字段耦合。
- `Theme.sourceSubmissionId` 只用于归因和奖励审计，正式主题展示一律读取 `Theme` 自己的字段。
- 正式 `Theme.generatorIdeas` 保存主题页和 Assistant 可直接使用的预设灵感；`avoidCliches` 保存该主题额外规避项。

当前业务表结构以 `prisma/schema.prisma` 和 `database/create-monica-ai.sql` 为准。本文只保留字段边界，避免技术方案和可执行 schema 分叉。

## 5. 状态机

### 5.1 Theme 发布与可见性

正式主题不使用 `status` 字段。

- 可见性由 `deleted = 0` 控制。
- 是否排期由 `publishDate` 是否为空控制。
- 是否今日主题或历史主题由 `publishDate` 和当前日期判断。

今日主题查询规则：

- `deleted = 0`
- `publishDate = today`
- 取 `publishDate` 最新的一条。

### 5.2 ThemeSubmission.status

- `under_review`：已提交，等待审核。
- `accepted`：管理员已接受为正式主题，并创建或关联 `Theme`。
- `not_selected`：未被采纳，可覆盖重复、质量不足、不安全或暂不适合等原因。

主题来源统一规则：

- 普通用户提交即创建 `ThemeSubmission(status=under_review)`，用户侧不需要草稿状态。
- 用户侧只展示 `Under review`、`Accepted`、`Not selected`。
- 管理员接受提交时创建正式 `Theme(sourceSubmissionId=themeSubmission.id)`，更新 `ThemeSubmission.status=accepted`，并把审核、奖励和主题关联节点写入 `reviewFlow`。
- 管理员直接创建主题时写入 `Theme`，不需要创建 `ThemeSubmission`。
- 普通用户不能直接创建 `Theme`；图片投稿不能自动创建 `Theme`。
- 积分奖励依据来自 `ThemeSubmission.status` 的变化，不新增积分模型，仍写入现有 `CreditAuditLog`。

### 5.3 GenerationJob.status

- `pending`：任务已创建。
- `safety_checking`：安全检查中。
- `queued`：等待调用供应商。
- `running`：供应商生成中。
- `succeeded`：生成成功。
- `failed`：生成失败。
- `blocked`：安全或额度原因阻止。
- `cancelled`：取消。

`GenerationJob.generationType`：

- `text_to_image`：当前版本的图片生成任务。

`generationType` 保留为任务类型扩展口，但当前不承载图片编辑、变体或来源图片设计。后续 AI 图片编辑需要重新设计任务表或独立业务表。

### 5.5 GeneratedImage.status

- `generated`：已生成，未投稿。
- `submitted`：已投稿，等待审核，用户不可删除或再次投稿。
- `approved`：投稿审核通过，并已创建公开投影，用户不可直接删除，只能申请撤销公开。
- `rejected`：投稿被拒。
- `withdrawal_requested`：用户已申请撤销公开，等待管理员审核。
- `withdrawn`：管理员同意撤销，公开记录已下架。
- `hidden`：系统或管理员隐藏，用户不可操作。

`GeneratedImage.isLocked` 是状态判断的冗余字段，用于用户侧和服务端快速判断是否允许删除、再次投稿等操作。只有 `generated`、`rejected` 对应 `isLocked=false`；其他状态都对应 `isLocked=true`。

### 5.6 ImageSubmission.status

- `submitted`：已提交，等待人工审核。
- `needs_review`：自动审核命中，需要人工复核。
- `approved`：审核通过，并创建 `PublicImage`。
- `rejected`：审核拒绝。

### 5.7 safetyStatus

- `pending`
- `passed`
- `failed`
- `needs_review`
- `skipped`

## 6. 后端服务设计

### 6.1 ThemeSubmissionService

职责：

- 创建用户提交的主题想法。
- 支持人工接受、编辑后接受或拒绝。
- 接受用户提交后创建正式 `Theme`，并保留提交记录用于归因。
- 触发主题提交相关积分奖励。

关键方法：

```ts
submitThemeIdea(userId, input): Promise<ThemeSubmission>
listMyThemeSubmissions(userId, input): Promise<PagedResult<ThemeSubmission>>
rejectThemeSubmission(reviewerUserId, input): Promise<ThemeSubmission>
acceptThemeSubmission(reviewerUserId, input): Promise<Theme>
createManagedTheme(adminUserId, input): Promise<Theme>
```

编辑规则：

- 用户提交后不可编辑，只能查看状态和 Notes。
- 管理员不修改 `ThemeSubmission.title/details/submitReason`，这些字段作为用户原始输入保存。
- 管理员接受时可以用提交内容初始化 `Theme.title/brief/themeNote`，但正式主题字段独立保存和编辑。
- 拒绝时更新 `ThemeSubmission.status=not_selected`，并把拒绝节点写入 `reviewFlow`。

### 6.2 ThemeService

职责：

- 获取今日主题。
- 查询主题列表。
- 查询主题详情。
- 查询主题作品。
- 维护主题统计。

关键方法：

```ts
getTodayTheme(): Promise<Theme | null>
listThemes(input): Promise<PagedResult<Theme>>
getThemeDetail(themeId: number): Promise<ThemeDetail>
listThemeImages(themeId: number, input): Promise<PagedResult<PublicImage>>
```

### 6.3 AssistantService

职责：

- Get Ideas。
- More like this。
- Improve Prompt。
- Ask Assistant。

AI 调用内部留空，通过 `AssistantProvider` 占位。
当前版本不持久化 Assistant 中间过程；Get Ideas、Improve Prompt、Ask Assistant 的输入输出只作为请求内返回和前端临时状态存在。用户最终点击 Generate 时，最终 prompt 写入 `GenerationJob.prompt`。

关键方法：

```ts
getIdeas(userId, input): Promise<IdeaResult>
moreLikeThis(userId, input): Promise<IdeaResult>
improvePrompt(userId, input): Promise<ImprovePromptResult>
ask(userId, input): Promise<AskAssistantResult>
```

### 6.4 GenerationService

职责：

- 计算生成消耗。
- 创建生成任务。
- 校验用户额度。
- 执行 prompt/reference 安全检查。
- 预扣减积分。
- 调用图片生成 Provider。
- 回写任务状态。
- 保存 GeneratedImage。
- 将 OpenRouter 返回的图片结果落到 R2。
- 生成成功时记录实际消耗。
- 生成失败、无结果、被拦截或超时时触发积分返还。
- 当前 `generationType = text_to_image`；字段保留为后续生成类任务扩展口。

关键方法：

```ts
estimateCost(input): Promise<CostEstimate>
createGenerationJob(userId, input): Promise<GenerationJob>
runGenerationJob(jobId): Promise<void>
syncGenerationJob(jobId): Promise<void>
```

积分规则：

- 创建任务前使用现有 credit service 检查余额。
- 创建任务时预扣减积分，避免用户并发生成时超额使用。
- `operationReferId = jobId`。
- `feature = ai_image_generation`。
- 任务成功时 `chargedCredits = estimatedCredits`。
- 任务失败、被拦截或取消时 `chargedCredits = 0`，由积分服务按 `jobId` 幂等返还预扣减积分。
- `GenerationJob` 不表达积分内部状态，积分服务通过 `CreditAuditLog.operationReferId` 负责幂等、返还和审计。

### 6.5 ImageService

职责：

- 查询图片详情。
- 校验图片归属。
- 创建下载 URL。
- 上传 reference image。
- 调用 R2 Explorer SDK 保存 reference image 和 generated image。
- 为 AI 调用生成可访问的 public/signed view URL。

关键方法：

```ts
getPublicImageDetail(publicImageId, viewerUserId?)
getMyImage(userId, imageId)
uploadReferenceImage(userId, input)
createProviderAccessibleImageUrl(referenceId): Promise<string>
storeGeneratedImageFromProviderResult(input): Promise<GeneratedImage>
```

R2 规则：

- `ReferenceImage` 只保存用户本地上传的参考图。
- `ReferenceImage.cdnImagePrefix` 保存 CDN 路径前缀或业务目录。
- `ReferenceImage.storageKey` 保存 R2 object key。
- `ReferenceImage` 不保存最终展示 URL；业务代码按 `IMAGE_CDN_BASE_URL + cdnImagePrefix + storageKey` 生成展示 URL，AI 调用前按需生成 Provider 可访问 URL。
- `GeneratedImage.sourceImageUrl` 保存 AI Provider 返回的原始图片 URL；如果该字段不为空，前端和服务端优先直接使用它。
- `GeneratedImage.cdnImagePrefix` 和 `GeneratedImage.storageKey` 保存生成图迁移到 R2 后的位置；当 `sourceImageUrl` 为空时，业务代码按 `IMAGE_CDN_BASE_URL + cdnImagePrefix + storageKey` 生成展示 URL。
- `GeneratedImage` 不保存缩略图 URL；如后续需要缩略图，单独设计派生资产或缓存策略。
- `GenerationJob` 不保存供应商原始响应；必要排查信息写入脱敏后的 `Apilog` 摘要。

### 6.9 SafetyService

职责：

- 为 AI 生成链路保留统一安全检查入口。
- MVP 阶段做轻量规则和 Provider 拒绝结果归一化。
- 后续可替换为自研 NSFW 审查、第三方审核、人工复核或 OpenAI moderation。
- 不把审核状态机和积分状态机塞进 `GenerationJob`。
- 投稿相关审核结果写入 `ImageSubmission.reviewFlow`，用于用户侧状态展示和管理端排查。

MVP 阶段不实现复杂自研 NSFW 分类器。当前默认策略是：

- Prompt：校验空内容、长度、明显违规关键词等基础规则。
- Reference image：校验文件类型、大小、归属、存储可访问性；图片内容审核入口保留，默认可放行。
- Provider result：默认信任 OpenRouter / 下游模型的安全策略；如果 Provider 明确因安全策略拒绝，归一化为 `blocked`。
- Submission：投稿进入公开展示前仍保留审核节点，MVP 可先走轻量自动审核或人工审核。

后续增强时只替换 `SafetyService` 实现，不改变 `GenerationService` 主流程、`GenerationJob.status` 语义和积分返还机制。

AI/审核供应商实现留空。

关键方法：

```ts
checkGenerationRequest(input): Promise<SafetyResult>
checkReferenceImages(input): Promise<SafetyResult>
checkProviderResult(input): Promise<SafetyResult>
checkSubmission(input): Promise<SafetyResult>
checkThemeRelevance(input): Promise<RelevanceResult>
```

`SafetyResult` 建议至少包含：

```ts
type SafetyResult = {
  status: 'passed' | 'blocked' | 'needs_review' | 'skipped';
  source: 'basic_rule' | 'provider_policy' | 'manual' | 'external_moderation';
  reasonCode?: string;
  detail?: Record<string, unknown>;
};
```

`skipped` 表示 MVP 阶段预留了检查点但当前没有启用复杂审核，不等同于系统不支持后续 NSFW 审查。

## 7. API 设计

### 7.1 鉴权级别

<CheetTable title="鉴权级别">

| 级别 | 说明 | 使用方式 |
| --- | --- | --- |
| Public | 不需要登录 | 不调用 `ApiAuthUtils` 或只读接口 |

### 7.4 Theme Submission API

#### `POST /api/theme-submissions`

鉴权：Required Auth

职责：

- 创建主题提交记录。
- 输入只包含 `title`、`details`、`submitReason?`。
- 创建 `ThemeSubmission(status=under_review)`。
- 如果是用户第一次提交主题，发放 +20 credits，并把奖励节点写入 `reviewFlow`。
- 提交后普通用户不可修改。

#### `GET /api/theme-submissions/my`

鉴权：Required Auth

职责：

- 查询当前用户提交过的主题想法。
- 返回用户侧状态：Under review、Accepted、Not selected。

#### `POST /api/admin/theme-submissions/:themeSubmissionId/review`

鉴权：Admin Required

路径参数：

- `themeSubmissionId` 是 `ThemeSubmission.id` 数字 ID。

- 人工审核主题想法。

- 拒绝时更新 `ThemeSubmission(status=not_selected)`，并把拒绝节点写入 `reviewFlow`。
- 接受时发放 +300 credits、发送站内信，并更新用户侧 Notes。

#### `POST /api/admin/themes`

鉴权：Admin Required

职责：

- 管理员直接创建正式主题。
- 写入 `Theme`，发布排期由 `publishDate` 表达，可见性由 `deleted` 表达。
- 支持编辑 `title`、`brief`、`themeNote`、SEO、tags、`generatorIdeas`、`avoidCliches`、`publishDate`、精选图。

## 10. 积分集成方案

### 10.1 消耗点

MVP 只在图片生成时消耗积分：

- Assistant 暂不消耗，或后续按配置开启。
- AI 图片编辑和变体当前不在 MVP 后端模型范围内；后续需要时重新设计。

### 10.2 消耗策略

1. `Estimate Generation Cost` 根据 model + imageCount 计算预估消耗。
2. `Create Generation Job` 事务中检查余额。
3. 创建 `GenerationJob(status=queued, estimatedCredits)`。
4. 调用现有 `creditService.consumeCredit` 或 reserve 能力做预扣减。
5. `operationReferId` 使用 `GenerationJob.jobId`。
6. `feature` 使用 `ai_image_generation`。

### 10.3 失败返还策略

需要区分失败类型：

- 安全检查失败：不创建供应商任务，返还预扣减积分。
- 余额不足：不创建任务或创建 `blocked` 任务，不扣积分。
- 供应商调用前失败：更新任务失败，返还预扣减积分。
- 供应商调用后无可交付结果、超时或返回错误：更新任务失败，返还预扣减积分。
- 返还失败不改变 `GenerationJob` 的任务语义，由积分服务重试或后台补偿保证最终成功。

`GenerationJob` 只记录任务状态和 `estimatedCredits / chargedCredits`。积分返还、流水幂等和审计由现有 credit service 与 `CreditAuditLog` 负责，不新增流水表。

## 11. 异步任务方案

异步任务优先使用已有 `@windrun-huaiin/backend-core` 的 Upstash / QStash 工具。QStash 负责把短请求变成可靠异步回调，前端仍以 `GenerationJob.status` 作为状态源轮询。

### 11.1 推荐方案：QStash FIFO Queue

- API 创建 `GenerationJob`。
- API 完成积分预扣减。
- API 使用 QStash FIFO queue 投递 `jobId`。
- QStash message body 只包含 `jobId`，不包含图片 base64、reference image URL 或 provider request。
- 返回 `jobId`。
- QStash 回调内部 Worker API。
- Worker API 校验 QStash 签名。
- Worker API 调用 `runGenerationJob(jobId)`。
- 前端轮询 `GET /api/generation/jobs/:jobId`。
- 建议初始 1-2 秒轮询一次，超过 30 秒后降频；收到 `succeeded/failed/blocked/cancelled` 后停止。

使用导入：

```ts
import {
  publishFIFOQueueMessage,
  verifyQstashSignature,
} from '@windrun-huaiin/backend-core/upstash/server';
```

发布任务：

```ts
await publishFIFOQueueMessage({
  queueName: 'monica_generation',
  url: `${process.env.NEXT_PUBLIC_BASE_URL}/api/internal/monica/generation/run`,
  body: { jobId },
});
```

不要在 QStash body 中放以下内容：

- 用户上传图片的 base64。
- R2 signed URL。
- OpenRouter provider request 全量参数。
- OpenRouter provider response。

这些内容都应由 Worker 根据 `jobId` 从数据库读取或临时生成。
如需排查供应商调用，只写入脱敏后的 `Apilog` 摘要或外部日志，不把原始 request/response 写入 `GenerationJob`。

QStash helper 会自动给 queue name 增加应用和环境前缀，实际 queue name 形如：

```text
{NEXT_PUBLIC_APP_NAME}_{live|test}_queue_monica_generation
```

### 11.2 Worker API

建议新增内部路由：

```text
POST /api/internal/monica/generation/run
```

职责：

- 读取 raw body。
- 使用 `verifyQstashSignature` 校验签名。
- 从 QStash envelope 中读取 `payload.jobId`。
- 调用 `GenerationService.runGenerationJob(jobId)`。
- 返回 2xx 表示本次消息处理完成。

QStash body 会被底层 helper 包一层 envelope：

```ts
type QstashEnvelope<TBody> = {
  source_msg_id: string;
  payload: TBody;
};
```

因此 Worker API 不应直接假设 body 根节点就是 `{ jobId }`，而应读取：

```ts
const { payload } = JSON.parse(rawBody) as QstashEnvelope<{ jobId: string }>;
```

签名校验：

```ts
const signature = request.headers.get('upstash-signature');
await verifyQstashSignature({
  signature,
  body: rawBody,
  url: request.url,
});
```

本地开发如需跳过签名，可使用：

```text
SKIP_UPSTASH_QSTASH_VERIFY=true
```

该开关只在 `NODE_ENV=development` 下生效。

### 11.3 幂等和重试

QStash 可能因为超时、网络错误或非 2xx 响应重试消息，所以 Worker 必须幂等。

必须保证：

- 如果 `GenerationJob.status` 已经是 `succeeded / failed / blocked / cancelled`，Worker 直接返回成功，不重复处理。
- 从 `queued` 更新到 `running` 时要带状态条件，避免并发 Worker 同时执行。
- AI mock 结果写入 `GeneratedImage` 时按 `jobId` 幂等，避免重复创建图片。
- OpenRouter 返回多张图片时，按 `jobId + providerImageIndex` 幂等写入。
- 失败返还由积分服务按 `operationReferId = jobId` 幂等处理。

Provider 原始 request/response、QStash message id、Worker 重试次数和锁信息不进入 `GenerationJob`。需要排查时写入脱敏后的 `Apilog` 或外部日志；如果未来确实需要业务级重试编排，再单独补字段或建任务执行日志表。

### 11.4 本地和 Mock 策略

MVP 如果本地没有 QStash 环境变量，`publishFIFOQueueMessage` 会返回 `null`。此时建议开发环境提供明确 fallback：

- `NODE_ENV=development` 时，可以直接 `setTimeout(() => runGenerationJob(jobId), 0)` 执行 mock worker。
- 生产环境不应静默 fallback，同步返回队列不可用错误或把任务标记为 `failed`。

这样可以保证本地开发便利，同时生产任务通路仍然可靠。

## 12. 数据查询策略

### 12.1 Explore 图片列表

查询 `PublicImage`：

- `deleted = 0`
- 按 `publishedAt`、`likeCount`、`featuredScore` 排序。
- 当前页 `imageId` 集合对应的 `GeneratedImage` 基础展示字段。
- 当前页 `themeId` 集合对应的 `Theme` 基础展示字段。
- 当前页作者信息按 `userId` 集合显式批量查询 `User` 基础展示字段。

登录用户额外查询：

- 当前页 `publicImageId` 集合对应的 `ImageLike`。
- 当前页 `publicImageId` 集合对应的 `ImageSave`。

避免 N+1。

### 12.2 Studio 图片列表

查询 `GeneratedImage`：

- `userId = currentUser.userId`
- `deleted = 0`
- 根据 Tab 映射 status。

Tab 映射：

- `All`：不限制 status，展示完整生成历史。
- `Submitted`：展示已投稿图片，并提供二级筛选。

Submitted 二级筛选：

- `Pending review`：`ImageSubmission.status = submitted`
- `Approved`：`ImageSubmission.status = approved`
- `Rejected`：`ImageSubmission.status = rejected`

### 12.3 Theme Gallery

查询 `PublicImage`：

- `themeId = theme.id`
- `deleted = 0`

排序：

- Featured：`featuredScore desc, publishedAt desc`
- Most liked：`likeCount desc, publishedAt desc`
- Newest：`publishedAt desc`

## 13. 安全与审核

### 13.1 安全检查点

安全检查点在主流程中固定保留，但 MVP 阶段可以使用轻量实现。

- Prompt 生成前检查：MVP 做基础规则；后续可接文本 NSFW / 有害内容审查。
- Reference image 上传后检查：MVP 做格式、大小、归属和可访问性校验；后续可接图片 NSFW / 暴力 / 版权风险审查。
- Provider 调用结果检查：MVP 默认信任 OpenRouter / 下游模型安全策略；如果 Provider 明确拒绝，任务归一化为 `blocked`。
- 生成图片完成后检查：MVP 可记录为 `skipped` 或 `passed`；后续可接输出图审核。
- 投稿时检查图片、prompt metadata、English title 和 creation note：公开展示前必须有审核入口，MVP 可先轻量自动审核或人工审核。
- 审核发布前检查主题相关性。

当前阶段不要把复杂 NSFW 审查前置成必选依赖。正确边界是：系统拥有稳定的 `SafetyService` 入口，MVP 使用轻量策略，投稿相关审核过程写入 `ImageSubmission.reviewFlow`。

Provider / OpenRouter 的安全策略只作为当前默认防线和补充防线，不作为 Monica 长期唯一的内容安全来源。

被安全策略拦截时：

- 未调用 Provider：`GenerationJob.status = blocked`，`chargedCredits = 0`。
- Provider 明确安全拒绝：`GenerationJob.status = blocked`，`chargedCredits = 0`。
- 已预扣减积分时，积分服务按 `GenerationJob.jobId` 幂等返还。
- `GenerationJob` 不记录审核状态机；投稿审核详情进入 `ImageSubmission.reviewFlow`。供应商原始响应不进入业务表。

### 13.2 reviewFlow 使用

`ImageSubmission.reviewFlow` 记录投稿自动和人工审核节点。MVP 阶段不是每个检查点都必须写完整节点；当检查结果影响投稿状态、公开展示或人工处理时，应追加节点。

`action` 示例：

- `safety_auto`
- `safety_manual`
- `theme_relevance`
- `publish_manual`

`result` 示例：

- `passed`
- `blocked`
- `needs_review`
- `skipped`
- `provider_rejected`
