## 1. 目标

本文用于收拢 Monica AI 的开发实施范围，按阶段推进前后端开发，并作为后续确认进度的依据。

当前结论：

- 除真实 AI Provider / OpenRouter 接入、统一手动联调后的小修外，MVP 前端主功能已经基本落地。
- 已完成：首页创作主流程、首页真实主题区、`/themes/[id]` 主题详情页、`/explore` Themes / Images 交互、`/studio` 图片管理和投稿弹窗、Monica 业务文案 i18n 拆分。
- 当前前端已经覆盖生成、轮询、参考图上传、结果展示、Studio 列表、状态筛选、查看 prompt、选择公开主题投稿、Explore 搜索/筛选/排序、图片详情、like/save 等主链路。
- 尚未产品化完成：真实 OpenRouter Provider、Download 实际能力、移动端和真实账号/积分/QStash/R2 的统一联调 polish。
- 因此当前状态应理解为“MVP 前端主链路完成，等待联调和真实 AI Provider 接入”，而不是最终产品功能全量完成。

详细 TODO:
  - OpenRouter 真实 AI Provider 还没接入，现在仍是 mock provider 链路
  - 管理端权限目前只是登录鉴权，是否真正 admin 权限需要接项目权限体系
  - like/save/view 的 Redis 实时计数 + 异步落库还没做，只是保留了字段和基础接口
  - Download 实际能力还没补
  - 撤销公开/下架申请审核流还没做
  - 真实 R2/QStash/账号/积分的端到端联调还没跑
  - 部分 Monica 新页面文案还是硬编码英文，后续要进 i18n

核心策略：

- 先打通“生成任务闭环”，再补 Studio、投稿、Explore 和真实 AI Provider。
- 异步任务从第一版开始使用真实 Upstash QStash 链路，不做 mock queue。
- Mock 只放在 AI Provider 层，用于模拟 OpenRouter 图片生成结果。
- `GenerationJob` 只表达任务状态和预计/实际积分消耗，不承载积分状态机和审核状态机。
- `GenerationJob.themeId` 只表示生成时使用过的主题上下文，可作为图片辅助信息；图片投稿时由用户在 Studio 选择一个当前公开主题，不强制绑定生成主题。
- Monica 业务表之间不声明 Prisma relation，只保留 scalar id 和索引；列表页由 repository/service 按 id 集合显式查询并组装展示数据。
- 内容安全保留 `SafetyService` 扩展入口，MVP 使用轻量策略，后续可替换为自研或第三方 NSFW 审查。
- 投稿、撤销和发布审核过程写入 `ImageSubmission.reviewFlow`；`GeneratedImage.status + isLocked` 决定用户是否还能删除、投稿或申请撤销。
- 正式 `Theme` 不使用独立 status 字段；公开可见条件为 `deleted=0` 且 `publishDate` 非空并不晚于当前产品日期，`publishDate=null` 表示未发布，不在 Explore 和公开主题详情中展示。

## 4. 阶段 2：真实 QStash 生成任务闭环

目标：不接真实 AI，先用真实 QStash webhook 链路跑通“创建任务 -> 入队 -> Worker -> mock AI -> 写入结果 -> 前端轮询”。

明确边界：

- 不做 mock 消息队列。
- 不做 `setTimeout` 本地 worker fallback。
- 本地开发也使用 Upstash QStash 线上 webhook 回调能力。
- Mock 只存在于 `ImageGenerationProvider` 实现层。

任务：

- 实现 `POST /api/monica/generation/jobs`。
- 实现 `GET /api/monica/generation/jobs/:jobId`。
- 实现 `POST /api/internal/monica/generation/run`。
- 使用 `publishFIFOQueueMessage` 投递 `{ jobId }`。
- 使用 `verifyQstashSignature` 校验 QStash 回调。
- Worker 从 QStash envelope 的 `payload.jobId` 读取任务 ID。
- 实现 `GenerationService.runGenerationJob(jobId)`。
- 实现 `MockImageGenerationProvider`。

生成任务创建流程：

1. 鉴权。
2. 参数校验。
3. 估算积分。
4. 创建 `GenerationJob(status=queued, estimatedCredits)`。
5. 预扣减积分，`CreditAuditLog.operationReferId = jobId`。
6. 投递 QStash FIFO queue，body 只包含 `{ jobId }`。
7. 返回 `jobId` 和 `queued`。

Worker 流程：

1. 校验 QStash 签名。
2. 读取 `payload.jobId`。
3. 幂等检查终态任务。
4. 条件更新 `queued -> running`。
5. 调用 `SafetyService.checkGenerationRequest`。
6. 调用 `MockImageGenerationProvider`。
7. 写入 `GeneratedImage`。
8. 更新 `GenerationJob(status=succeeded, chargedCredits=estimatedCredits)`。
9. 失败、blocked、provider rejected、无结果或超时时，`chargedCredits=0`，由积分服务按 `jobId` 幂等返还。

验收标准：

- 创建任务 API 能返回 `jobId`。
- QStash 能真实回调 Worker API。
- 轮询接口能看到 `queued -> running -> succeeded`。
- 重复 Worker 回调不会重复创建图片或重复扣积分。
- `failed / blocked / provider_rejected` 路径能返还积分。

## 5. 阶段 3：R2 图片上传和 Reference Image

目标：用户参考图可以上传到 R2，并通过 `referenceId` 参与后续生成。

任务：

- 实现 `POST /api/monica/reference-images`。
- 使用 `src/lib/r2-explorer-sdk.ts` 上传图片到 R2。
- 保存 `ReferenceImage.cdnImagePrefix`、`storageKey`、mime type、width、height；不在 DB 固化最终展示 URL。
- 返回 `referenceId` 和预览 URL。
- 生成任务支持 `referenceId`；Worker 根据 `referenceId` 查库，用 `NEXT_PUBLIC_CDN_ACCESS_DOMAIN + cdnImagePrefix + storageKey` 拼接 Provider 可访问 URL。
- 后续如 CDN 访问需要授权，再基于同一 `storageKey` 增加临时访问 URL 能力。
- QStash body 仍然只传 `{ jobId }`。

验收标准：

- 上传图片后前端可以预览。
- 生成任务只传 `referenceId`，不传图片 base64 或 signed URL。
- Worker 能根据 `jobId` 查到 `ReferenceImage` 并按 CDN 访问域名拼接 Provider 可访问 URL。
- `SafetyService.checkReferenceImages` 调用点存在，MVP 可返回 `skipped`。

## 6. 阶段 4：前端创作主流程

目标：完成用户可用的创作体验，接入真实后端任务状态和 mock AI 结果。

任务：

- 创作输入区：
  - prompt。
  - model。
  - style。
  - aspect ratio。
  - image count。
  - reference image 上传。
  - 预估积分展示。
- Generate 提交流程：
  - 调用创建任务 API。
  - 进入任务进行中状态。
  - 禁用重复提交。
- 实现 `useGenerationJobPolling`。
- 展示状态：
  - `queued`
  - `running`
  - `succeeded`
  - `failed`
  - `blocked`
  - `cancelled`
- 成功后展示生成图片。
- 失败或拦截时显示重试入口和不消耗积分语义。

样式边界：

- 参考 `src/components/hero.tsx` 的视觉语言。
- 使用 `docs/skills` 中现有组件规范。
- 除 `slot-text` 外，优先复用 `windrun-huaiin*` 底层组件。
- 第一屏应该是可创作体验，不做营销 landing。

验收标准：

- 桌面和移动端创作流程可用。
- 轮询在终态后停止。
- `failed / blocked / cancelled` 不展示为成功结果。
- 前端展示后端返回的图片 URL；mock 或真实 provider 原始 URL 可先保存为 `sourceImageUrl`，真实 Provider 临时 URL 后续需要迁移到 R2 后保存 `cdnImagePrefix/storageKey`。

## 7. 阶段 5：Studio / 投稿 / Explore

目标：把生成图片接入产品闭环，区分私有图片和公开展示。

任务：

- My Studio：
  - 我的生成图片列表。
  - 状态筛选。
  - 查看 prompt。
  - 下载。
  - 根据 `GeneratedImage.status` 和 `isLocked` 控制 Submit、删除、申请撤销等操作。
  - Submit 弹窗允许用户选择一个当前公开主题；生成时的主题只作为默认选项或辅助信息，不是投稿前置条件。
- 投稿：
  - `POST /api/monica/submissions`。
  - 请求包含 `imageId`、用户选择的 `themeId`、公开标题和创作说明。
  - 创建 `ImageSubmission`。
  - 追加 `ImageSubmission.reviewFlow` 审核节点。
  - 提交后更新 `GeneratedImage(status=under_review, isLocked=true)`。
  - MVP 可先使用轻量自动审核或人工审核。
  - 审核通过时创建 `PublicImage(deleted=0)`，更新 `GeneratedImage(status=approved, isLocked=true)`。
  - 审核拒绝时更新 `GeneratedImage(status=rejected, isLocked=false)`。
  - 已公开作品不允许用户直接删除底层图片；用户只能申请撤销公开，后续由管理员审核后设置 `PublicImage.deleted=1` 并更新 `GeneratedImage(status=withdrawn, isLocked=true)`。
- Explore / Theme Gallery：
  - Themes 只展示已发布正式主题，条件为 `Theme.deleted=0`、`publishDate` 非空且不晚于当前产品日期。
  - 主题列表和管理员主题列表展示 `coverImageUrl` 和最多前三张 `theme_featured_images` 对应的公开图片。
  - 只展示 `PublicImage`。
  - 实现 like/save。
  - 实现 view 和 unique view 聚合计数。
  - like/save/view 计数先由 Redis 实时维护，异步批量落库到 `PublicImage` 聚合字段。
  - 支持 newest、most liked、featured 排序。
  - 列表页按 `imageId`、`themeId`、`userId` 集合显式批量查询展示所需基础数据。

验收标准：

- 私有 `GeneratedImage` 不会直接公开。
- Studio 中未锁定且状态为 `generated/rejected` 的图片可以选择任意公开主题投稿。
- 只有投稿审核通过后才创建 `PublicImage`。
- `PublicImage` 不使用 status，公开可见条件为 `deleted=0`。
- `GeneratedImage.isLocked` 与状态一致：`generated/rejected=false`，`under_review/approved` 等审核或公开状态为 `true`。
- 点赞和收藏幂等。
- Redis 计数异步落库后，`PublicImage.likeCount/saveCount/viewCount/uniqueViewCount` 可用于排序和兜底展示。
- 投稿状态能在 Studio 中查看。
- 撤销公开申请能进入审核流，用户不能绕过审核直接删除已公开图片。

## 8. 阶段 6：OpenRouter 真实 Provider

目标：用真实 OpenRouter 替换 `MockImageGenerationProvider`，不改变业务主流程。

任务：

- 实现 `OpenRouterImageGenerationProvider`。
- 支持文生图。
- 支持图文生图。
- 支持多图输出。
- 解析 `message.images[]`。
- 处理 base64 data URL 输出：
  - 解析 mime type。
  - 转二进制。
  - 上传 R2。
  - 创建 `GeneratedImage`。
- 归一化 Provider 错误：
  - 超时。
  - 无结果。
  - 安全拒绝。
  - 参数不支持。
  - 模型不可用。

验收标准：

- 文生图真实生成成功。
- 图文生图真实生成成功。
- 一次多图输出能创建多条 `GeneratedImage`。
- Provider 安全拒绝映射为 `provider_rejected` 或 `blocked`，不消耗积分。
- Provider 失败、超时、无结果路径能返还积分。

## 9. 阶段 7：硬化和回归

目标：补齐上线前必须的可靠性、权限和体验检查。

任务：

- Worker 幂等：
  - 终态任务直接返回 2xx。
  - `queued -> running` 带状态条件。
  - `jobId + providerImageIndex` 防重复图片。
- 积分幂等：
  - 预扣减只执行一次。
  - 返还按 `operationReferId = jobId` 幂等。
  - 返还失败有补偿或后台重试入口。
- 权限：
  - 用户只能查询自己的任务和私有图片。
  - PublicImage 才能匿名公开访问。
  - 内部 Worker API 只接受 QStash 签名请求。
- 查询：
  - 不使用 Prisma relation include 串联 Monica 业务表。
  - 列表页按 id 集合批量查询并在 service 层组装。
- 审核：
  - `ImageSubmission.reviewFlow` 覆盖提交、自动审核、人工审核、发布、撤销申请和撤销审核节点。
  - `GeneratedImage.isLocked` 防止锁定状态下的删除、重复投稿和绕过撤销审核。
- 前端：
  - 移动端布局检查。
  - 长 prompt 和长错误信息不撑破 UI。
  - 任务状态切换不闪烁。
- 观测：
  - 记录 `GenerationJob.failureCode`。
  - 通过脱敏 `Apilog` 或外部日志记录 provider 错误码、耗时、状态码等摘要。
  - 记录 QStash message id。

验收标准：

- 主要 happy path 和失败路径都有手动验证记录。
- 构建、type check、lint 或项目既有检查通过。
- 异步任务重复投递不会造成重复扣费或重复图片。
