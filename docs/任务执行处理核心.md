# 任务执行处理核心

本文记录 Monica 图片生成任务的现有执行逻辑、积分回滚事务的边界，以及失败结算在重试场景下仍需要完善的部分。

## 一、当前任务执行流程

任务创建时，服务在一个数据库事务中完成以下动作：

1. 创建 `generation_jobs`，初始状态为 `queued`。
2. 创建任务与参考图关联。
3. 根据模型和图片数量计算 `estimated_credits`。
4. 消费可用积分，并写入 `credit_audit_log(operation_type = 'consume', operation_refer_id = jobId)`。

队列 worker（或 `client-run`、`inline` 模式）执行 `runGenerationJob(jobId)`：

```text
queued
  -> 原子 claim 为 running
  -> 请求安全检查
  -> 调用图片供应商
  -> 检查供应商结果
  -> 上传图片到 R2
  -> 写入 generated_images，任务更新为 succeeded
```

正常成功时，`generation_jobs.charged_credits` 写为 `estimated_credits`。创建任务时已经实际扣费；`charged_credits` 是任务结果字段，不是第二次扣费。

当前失败来源包括：

- QStash 队列发布失败；
- 请求安全检查拦截；
- 图片供应商调用失败或无可交付图片；
- 供应商结果安全检查拦截；
- R2 上传失败；
- 生成成功落库失败等运行期异常。

## 二、当前失败收尾事务

失败或拦截后，代码进入统一的 `finalizeUnsuccessfulJob(...)`。该方法使用 backend-core 的 `runInTransaction`，失败时会记录统一的 `TRANSACTION ROLLBACK` 日志、操作名、耗时、错误及错误码。

事务内的顺序为：

```text
1. SELECT generation_jobs WHERE job_id = ? FOR UPDATE
   锁定该任务行，串行化同一 jobId 的失败结算。

2. 读取 generation_jobs.credits_rolled_back_at
   非空：表示此前已成功结算，直接结束，不再次返还。

3. 更新 generation_jobs
   status = failed / blocked
   charged_credits = 0
   failure_code / failure_message / completed_at = 对应失败信息

4. 查询同一 jobId 的 consume 审计流水
   分别汇总 free / paid / one_time_paid 三个积分桶。

5. 调用 rollbackConsumeCredit(..., tx)
   只恢复 credits.balance_*；
   不变更 total_*_limit；
   写入 adjust_increase 审计流水，operation_refer_id = jobId。

6. 更新 generation_jobs.credits_rolled_back_at = now()

7. 提交事务
```

`credits_rolled_back_at` 是生成任务的业务幂等标记；`credit_audit_log` 只承担审计和追溯职责，不承担“只允许返还一次”的唯一约束职责。

## 三、事务回滚时的真实结果

事务中的 SQL 虽按上述顺序执行，但未提交前对其他数据库连接不可见。

如果第 3 到第 6 步的任意操作失败，整个事务回滚，数据库会恢复到进入该事务前的状态：

```text
generation_jobs.status                  保持事务开始前的值（通常为 running）
generation_jobs.credits_rolled_back_at  保持 NULL
credits.balance_*                        保持已扣费后的值
credit_audit_log                         不会留下本次 adjust_increase 回滚流水
```

因此不会出现“余额已返还但任务仍 running”或“任务显示失败、但余额没有返还”的半完成状态。这是事务必须保持的原子性。

但是，回滚不等于后续一定会自动恢复。事务异常会向外抛出：

- QStash worker route 会返回 HTTP 500；是否重新投递取决于 QStash 队列配置。当前入队调用没有显式传入 retries 配置。
- `client-run` 模式中，客户端或用户只有再次请求 run endpoint 才可能触发后续执行。
- `inline` 模式中，请求直接失败，不存在独立 worker 自动重试。

若没有后续重试或后台补偿，任务会停在 `running`，积分保持已扣减状态。

## 四、当前重试策略的缺口

当前 `runGenerationJob` 对 `running` 状态的处理存在风险：当重复 worker 未能从 `queued` claim 到任务，但读取到任务仍为 `running` 时，它不会立即退出，而会继续执行后续生成流程。

这在“生成已失败，但失败结算事务又回滚”的场景会造成问题：

```text
第一次 worker
  供应商调用失败
  -> 尝试失败结算事务
  -> 数据库故障，事务回滚
  -> job 仍为 running
  -> worker 返回 500

QStash / 客户端重试
  -> 看到 job 为 running
  -> 当前实现可能再次调用供应商，而不是只重试积分结算
```

可能的副作用：

- 再次调用图片供应商，产生额外供应商成本；
- 若供应商首次其实已接受请求但网络响应失败，可能生成重复图片；
- 再次上传 R2，产生重复对象或覆盖/并发写入风险；
- 任务最终结果可能与第一次外部调用不一致；
- 虽然创建任务的 `consume` 流水可防止第二次扣积分，但“外部资源调用一次”的语义无法保证；
- 如果重试耗尽，任务持续停留在 `running`，需要人工处理。

根因是：数据库可以原子化“任务状态、积分余额、审计流水”，但不能把供应商调用、R2 上传等外部副作用纳入同一个 PostgreSQL 事务。不能用“重跑整个任务”来代替“重试失败结算”。

## 五、建议的两阶段失败结算方案

应把“记录外部执行失败”和“积分回滚结算”拆成可恢复的两个阶段，并让队列重试根据持久化状态只执行尚未完成的阶段。

建议新增两个非终态状态：

```text
failed_refund_pending
blocked_refund_pending
```

或使用等价的 `failure_pending_refund_at` / `failure_outcome` 字段。关键要求是：数据库中必须能明确表示“生成不会再调用供应商，但积分回滚尚未完成”。

推荐状态流转：

```text
queued -> running -> succeeded

queued/running -> failed_refund_pending
failed_refund_pending -> failed（积分已回滚）

queued/running -> blocked_refund_pending
blocked_refund_pending -> blocked（积分已回滚）
```

### 阶段 A：持久化失败意图

生成、安全检查或上传发生失败后，执行一个很短的数据库事务：

```text
锁定 job 行
写入 failed_refund_pending / blocked_refund_pending
写入 failure_code、failure_message、失败发生时间
提交
```

此阶段不返还积分，也不调用任何外部服务。其目的不是最终结算，而是让任何重试都能知道：该任务的外部生成已结束，不应再调用供应商。

### 阶段 B：短事务完成积分结算

随后执行当前已有的结算事务：

```text
锁定 job 行
确认处于 *_refund_pending 且 credits_rolled_back_at 为 NULL
读取 consume 审计流水并按桶汇总
rollbackConsumeCredit(..., tx)
写 adjust_increase 审计流水
写 credits_rolled_back_at
将状态改为最终 failed / blocked
提交
```

阶段 B 失败时，状态保持 `*_refund_pending`，余额仍保持扣减状态；队列重试或后台补偿只执行阶段 B，不会再次调用供应商或上传 R2。

## 六、Worker 的重试分派规则

`runGenerationJob(jobId)` 应根据数据库状态分派，而不是仅将 `running` 视为可继续执行：

| 当前状态 | worker 行为 |
| --- | --- |
| `queued` | 尝试 claim；成功后执行完整生成流程。 |
| `running` | 不再次执行供应商调用；返回“已有执行中”，或由超时恢复机制单独处理。 |
| `failed_refund_pending` | 仅执行阶段 B 积分结算。 |
| `blocked_refund_pending` | 仅执行阶段 B 积分结算。 |
| `succeeded` / `failed` / `blocked` / `cancelled` | 终态，直接返回。 |

对于卡死在 `running` 的任务，应有单独的超时恢复任务，例如根据 `started_at` 超过执行上限、无心跳或无 provider job 状态来决定转为 `failed_refund_pending`。不能由任意重复投递直接重跑。

## 七、外部供应商的进一步幂等要求

两阶段失败结算能防止“结算失败后重复调用供应商”，但无法完全消除“供应商已收到请求、应用在收到响应前中断”的不确定窗口。

应继续采取以下措施：

1. 将 `jobId` 作为供应商请求的 idempotency key；若供应商支持幂等键，重试必须传同一键。
2. 尽早持久化 `providerJobId` / provider request id；恢复任务优先查询供应商任务结果，而非重新创建生成请求。
3. 为 `running` 任务记录执行尝试时间、尝试次数、最后心跳或租约到期时间，供超时恢复任务判断。
4. R2 对象 key 应固定为 `ai/{jobId}/{imageIndex}.{ext}`，使同一任务的重试具备可预测的写入位置；仍需处理不同响应产生不同图片索引或 metadata 的情况。

## 八、日志与监控要求

- 所有写入型事务均通过 `runInTransaction` 执行，并传入稳定的操作名；失败时由底层统一打印事务回滚日志。
- 生成执行失败应记录外部失败详情（供应商、阶段、请求摘要、错误摘要）。
- 失败结算事务失败应额外有可检索的告警或 `apilog`，至少包含 `jobId`、目标状态、failureCode、错误码和重试次数。
- 后台应监控长期 `running`、长期 `*_refund_pending`、以及 `credits_rolled_back_at IS NULL` 的失败任务。
- 用户界面只有在任务最终进入 `failed` / `blocked` 且积分已结算后，才展示“积分未被使用/已返还”；`*_refund_pending` 应展示“任务失败，正在处理积分返还”。

## 九、当前实现与目标的差异

当前实现已经具备：

- 同一任务失败结算时的 `FOR UPDATE` 行锁；
- 积分余额、回滚流水、任务最终状态和 `credits_rolled_back_at` 的原子提交；
- 回滚只恢复 `balance_*`，不增加 `total_*_limit`；
- backend-core 统一事务回滚日志。

当前尚未具备：

- `failed_refund_pending` / `blocked_refund_pending` 等可恢复结算状态；
- 重试时只重试积分结算、禁止重跑供应商调用的分派逻辑；
- `running` 超时恢复与补偿 worker；
- 供应商请求的持久化幂等键及 provider job 查询恢复；
- 失败结算失败的专门告警与后台补偿入口。

在补齐这些能力前，数据库事务能保证“返还要么完整提交、要么完全不发生”，但不能单独保证“外部生成只执行一次”或“返还最终一定完成”。
